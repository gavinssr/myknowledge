开源项目贡献指南

无构建步骤的 HTML/CSS/JS 前端项目 — 皮肤、主题、纯样式 PR 通用流程

nesquena/hermes-webui 实操提炼

流程概览

  1. 保存本地定制 · 创建干净分支
  2. 在新分支上应用要贡献的改动
  3. 启动项目 · 浏览器验证
  4. 测试中发现问题 → 定位原因 → 修改 → 再测
  5. 预检 + 提交
  6. Fork 仓库 + 推送 + 开 PR
  7. 恢复本地完整定制

保存本地定制 · 创建干净分支

目标：你的本地项目上有一堆自己用的改动（比如额外的按钮、自定义图标、本地优化），但你要贡献给上游的只是其中一个独立功能（比如一个新皮肤）。第一步就是把所有改动"寄存"起来，从干净起点开始。

cd ~/your-project
git stash push -m "all local customizations"
git checkout -b feat/my-feature
git stash push -m "…”
把工作区所有未提交的改动（已修改文件 + 新增文件）打包进一个临时存储区，工作区恢复到和最新 commit 完全一致。你的改动没有丢，只是"寄存"了。

git checkout -b feat/my-feature
同时创建并切换到新分支。-b = git branch + git checkout 的快捷方式。PR 必须来自独立分支，不能直接在 master 上提交。
常见异常
如果终端提示 error: Your local changes to the following files would be overwritten by checkout，说明有改动没 stash。先执行 git stash 再重试。

在新分支上应用要贡献的改动

本项目改动了 4 个文件：

static/style.css
+46 行，Hepburn 皮肤 CSS
static/boot.js
+1 行，皮肤注册
static/index.html
+1 行，内联白名单
static/panels.js
+2/-2 行，下拉框回显修复

操作方式 — 用你最顺手的编辑器打开文件修改。纯 CSS/JS 项目没有构建步骤，改了直接生效。

关键原则：只带要贡献的内容
你的本地定制（shutdown 按钮、自定义图标等）不要带进 PR。如果某个文件既有皮肤改动又有本地改动，只取皮肤那部分。修改前先看清楚哪些行是皮肤、哪些是你的私有功能。

启动项目 · 浏览器验证

./ctl.sh restart          # 或你的项目的启动命令

浏览器打开页面，逐项检查：

  • 核心功能：皮肤选择后颜色正确渲染
  • 边界情况：Light / Dark / System 三种主题模式下都正常
  • 入口一致性：设置面板、slash 命令（如果有的话）都能选中皮肤
  • 持久化：刷新页面后皮肤保持不丢失

测试中发现问题 → 定位 → 修复 → 再测

本次遇到的实际 bug：选 Hepburn 后页面渲染正确，但设置面板下拉框仍显示 "default"。

定位方法：

  1. 打开 DevTools → Elements，确认 <html data-skin="hepburn"> 已生效 → 皮肤确实应用了
  2. 问题缩小到设置面板的 UI 控件本身没同步
  3. 在 panels.js 里搜索 skinVal，找到设置面板读取皮肤值的逻辑
  4. 发现它只读服务端返回值 settings.skin（默认 'default'），忽略了用户实际选择

修复：一行改动，让下拉框优先读 localStorage：

// 改前 — 只读服务端
const skinVal=(settings.skin||'default').toLowerCase();

// 改后 — localStorage 优先
const skinVal=(localStorage.getItem('hermes-skin')||settings.skin||'default').toLowerCase();

再测：

./ctl.sh restart    # 重启加载新代码
# 浏览器 Cmd+Shift+R 硬刷新，确认下拉框正确显示 "Hepburn"
通用排查思路
渲染正确但 UI 控件不对 → 不是数据流问题，是 UI 同步问题 → 搜索下拉框的 value 赋值代码 → 看它从哪个数据源取值 → 改数据源优先级。

预检 + 提交

分两段执行，中间有检查点：

第一段 — 暂存 + 确认范围：

git add static/style.css static/boot.js static/index.html static/panels.js
git diff --cached --stat
输出示例： 
 static/boot.js    |  1 +
 static/index.html |  2 +-
 static/panels.js  |  2 +-
 static/style.css  | 46 ++++++
 4 files changed, 49 insertions(+), 2 deletions(-)

第二段 — 确认无误后提交：

git commit -m "feat: add Hepburn skin (magenta-rose palette)

Add Hepburn skin with full light/dark palette derived from the
Hepburn TUI theme. Brand color #c6246a with pink-magenta accents.

Also fix settings panel skin picker to prioritize localStorage
over server defaults."
git diff --cached --stat
--cached 看的是暂存区（git add 之后的状态），不是工作区。--stat 只显示文件名和增删行数。

为什么分两段
add 完先看一眼确认没有多带或少带文件，再 commit。多带文件（比如你的私有改动被误 add）→ 立刻 git reset 纠正。少带文件 → 补 git add。
Commit message 规范
第一行用 feat: 或 fix: 前缀，后面跟简短描述（英文 50 字以内）。空一行后写详细说明。大多数开源项目期望这样，方便自动生成 changelog。

Fork 仓库 + 推送 + 开 PR

6a · Fork 仓库

浏览器打开上游仓库（如 github.com/nesquena/hermes-webui）→ 右上角 Fork → Create fork。仓库名不改，描述不填，全用默认。

6b · 添加 remote + 推送

git remote add myfork https://github.com/你的用户名/hermes-webui.git
git push myfork feat/my-feature
为什么叫 myfork 而不是 origin
origin 保持指向上游仓库（你没有 push 权限），myfork 指向你的 fork（你有完全权限）。以后 git pull origin master 同步上游更新，git push myfork 推送你的 PR，两不干扰。

6c · 开 PR

推送成功后终端会输出快捷链接，浏览器打开。关键设置：

  • Base repository → 上游仓库（不是你的 fork）
  • Base branch → master
  • Head repository → 你的 fork
  • Compare → feat/my-feature

PR 描述结构（按项目 CONTRIBUTING.md 要求）：

## Thinking Path
（从项目目标到你做的具体改动，几步推理链）

## What Changed
（列出改动的文件和内容）

## Verification
（你怎么验证的，附截图）

## Model Used
（用了什么 AI 辅助，或者写 None — human-authored）
如果 push 被拒绝
remote: Permission to nesquena/hermes-webui.git denied
说明你 push 到了 origin 而不是 myfork。重新执行 git push myfork feat/my-feature 即可。对 origin 的 push 被拒绝，没有副作用。

恢复本地完整定制

git checkout master
git stash pop
./ctl.sh restart
git stash pop
把第一步寄存的改动还原到工作区，同时删除 stash 条目。你的 shutdown 按钮、自定义图标等本地功能全部回归。
如果 stash pop 有冲突
CONFLICT (content): Merge conflict in static/boot.js
原因：上游 master 在你 stash 之后有了新提交，且新提交改了你 stash 里也改过的同一个文件的同一行。

处理：手动编辑冲突文件，找到 <<< / === / >>> 标记的行，保留需要的版本，删除标记。然后 git add 冲突文件。如果不想手动处理，git reset --hard 放弃 pop（stash 不会被删除，可稍后再试）。

指令速查表

指令 作用 关键参数
git stash push 寄存所有未提交改动，清空工作区 -m 加备注
git stash pop 还原最近一次 stash 的改动并删除记录 无
git checkout -b 创建并切换到新分支 分支名
git add 将文件加入暂存区 文件名（多个空格分隔）
git diff --cached --stat 预览暂存区改动摘要 无
git commit -m 将暂存区提交为一个 commit 消息字符串
git remote add 添加远程仓库别名 别名 + URL
git push 推送分支到远程仓库 远程别名 + 分支名
git reset --hard 丢弃工作区和暂存区所有改动（危险） 无
适用项目类型
本指南适用于 "无构建步骤 + 纯前端文件在项目仓库中直接可改" 的开源项目。典型特征：没有 webpack.config.js、没有 package.json 里的 build 脚本、静态文件是明文 HTML/CSS/JS 而非打包产物。如果你的项目是 Vue/React SPA（有 build 步骤），皮肤/主题贡献需要在源码层修改后重新构建，流程不同。
